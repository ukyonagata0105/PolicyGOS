# 指標抽出改善計画

## 背景

現状の実装では、政策評価資料から抽出した指標を `activity` と `outcome` の二種類に強く寄せて保持している。これは UI 上で「何を実施したか」と「どのような成果があったか」を見せるうえでは分かりやすいが、実データ側の多様性を十分に受け止められていない。

実際の自治体資料には、活動指標と成果指標だけでなく、進捗指標、執行指標、KPI 的な中間指標、目標そのもの、定性的な評価項目などが混在する。これらを `activity` または `outcome` に無理に押し込むと、誤分類や欠落が起きやすくなる。また、資料内で指標種別の区別が明示されているのに、その区別を失ったまま canonical 化すると、後続の UI や比較分析で再利用しにくくなる。

したがって、今後の方針は「最初に分類を固定する」のではなく、**まず指標を回収し、その後に分類する** に切り替える必要がある。

## 方針

今後の基本方針は次の三点に置く。

1. 指標らしいものは、取れる限りすべて回収する  
2. 資料中に指標の種類が明示されている場合は、その種類ごとに分けて保持する  
3. 種類が判定できない場合でも、捨てずに `unclassified` として保持する  

つまり、抽出段では「回収優先」、整形段では「分類可能なら分類」、UI 段では「主要種別を見せつつその他も失わない」という構成を採る。

## 現状の問題

現在の `IndicatorRecord` は `activity | outcome` 前提であり、row から指標が見つかっても、この二種類に対応しないものは取りこぼす可能性がある。また、row から見つかった値のうち、どの列を根拠にその指標を構成したのかを十分に残していないため、後から再分類や修復を行うコストが高い。

加えて、現在の評価では「活動指標が 1 件あれば最低限通る」状態になっており、成果指標やその他指標が回収できていなくても E2E 上は完走する。このため、完走と抽出品質が切り離されており、資料ごとの指標の豊富さが UI に十分反映されていない。

## 目標状態

指標抽出の目標状態は、各事業について以下のような情報を保持できることである。

- `label`
  資料上の指標名
- `kind`
  `activity | outcome | progress | budget | kpi | qualitative | unclassified`
- `raw_kind_label`
  資料中での見出しや列ラベル
- `unit`
- `planned_value`
- `actual_value`
- `target_value`
- `achievement`
- `source_refs[]`
- `supporting_fields[]`
- `confidence`

このとき `kind` は最初から必ず決め打ちしなくてよい。資料上で `活動指標` `成果指標` が明示されていればそれを使い、明示がなければ `unclassified` として保持し、後続の Gemini 判定または deterministic rule で補助的に分類する。

## 実装方針

### 1. 指標型の拡張

`IndicatorType` は `activity | outcome` 固定から拡張する。最小限、以下を持つ。

- `activity`
- `outcome`
- `progress`
- `budget`
- `kpi`
- `qualitative`
- `unclassified`

ただし v1 で最重要なのは、分類種類を増やすことそのものではなく、**活動・成果以外を捨てないこと**である。

### 2. row からの指標回収を優先する

`buildIndicatorsFromRowFields()` は現在の prefix ベース抽出だけでなく、以下の順で回収を行う。

1. 明示的な `活動.*指標` / `成果.*指標` を優先抽出
2. `進捗` `KPI` `評価指標` `目標指標` `成果目標` などの列ラベルを拾う
3. `指標` を含むが種類不明な列は `unclassified` として回収
4. 値列は `計画`, `当初`, `現状`, `実績`, `目標`, `達成` との対応で束ねる

この処理では、見つかった指標を一件だけ返すのではなく、**複数件返す** 前提に変える。

### 3. raw 情報を落とさない

各指標に対して、次を保持する。

- どの row field を根拠に使ったか
- 元の列名
- 元の値
- source reference

これにより、後続で `repair-runner` や Gemini に再判定させる際にも「何を根拠に組み立てたか」が失われない。

### 4. Gemini の役割を「再分類」に寄せる

Gemini は指標そのものを一から自由生成するのではなく、まず回収済みの candidate indicators を見て、

- この指標は `activity` か `outcome` か
- それ以外のどの種別に近いか
- `raw_kind_label` と `label` のどちらを優先すべきか

を判定する補助に使う。

つまり、指標抽出でも同様に、

- deterministic code: 取れるものを全部拾う
- Gemini: 種別判定や列対応の補助
- quality gate: 危険なものを止める

という責務分離を維持する。

### 5. UI は「主要分類」と「その他指標」を分けて見せる

`Project Explorer` と `Briefing` では、従来どおり `活動指標` と `成果指標` を上位表示してよい。ただし、それ以外の指標が取れた場合は `その他指標` として確認可能にする。

これにより、市民向け UI の単純さを保ちつつ、データ損失を防ぐことができる。

## 修復レーンとの関係

`repair-runner` は今後、指標についても次を行えるようにする。

- 欠落した成果指標の補完
- `unclassified` 指標の再分類
- `department` と `status` のような非指標フィールドとの取り違え修正
- 活動指標と成果指標の入れ違い修正

ただし修復レーンの役割は「指標を新たに創作する」ことではなく、**回収済み候補を正しく再配置すること**に置く。

## テスト方針

最低限、以下のケースを追加する。

1. `活動指標` と `成果指標` が両方ある row で両方取れる  
2. `評価指標` だけで種類不明な row で `unclassified` として取れる  
3. `活動` でも `成果` でもない複数指標が同じ row にある場合に複数件返せる  
4. 奥州市の row-oriented PDF で `activity` だけでなく `outcome` 候補が取れるケースを fixture 化する  
5. 岩手県の `1-7 被災地こころのケア対策事業` で活動指標・成果指標が保持される  

## 段階的な実装順序

### Phase 1

- `IndicatorType` を拡張する
- `buildIndicatorsFromRowFields()` を複数指標返却にする
- `unclassified` を捨てずに保持する

### Phase 2

- `repair-runner` に指標再分類の prompt を追加する
- `activity/outcome` の confidence を改善する
- `supporting_fields` を UI / debug に見せる

### Phase 3

- `Project Explorer` に `その他指標` 表示を追加する
- `Briefing` にも必要な場合だけ補助指標を出す
- 複数自治体の資料で指標種別クラスタを比較できるようにする

## 結論

今後の指標抽出で重要なのは、「活動指標か成果指標かを最初に決めること」ではない。重要なのは、**指標として読み取れるものをまず失わず回収し、その後に分類すること**である。

この方針に切り替えることで、奥州市のような row-oriented 文書だけでなく、他自治体の異なる様式でも指標情報を落としにくくなり、UI 側でも「主要指標」と「その他指標」を使い分けた表現が可能になる。
