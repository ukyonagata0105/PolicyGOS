# PolicyEval GOS 開発経過報告

## 1. 本報告書の位置付け

本報告書は、`PolicyEval GOS` の開発経過について、当初想定した実施内容と実際の試行結果との差異を整理し、今後の改善方針を明らかにすることを目的とするものである。対象事業の趣旨は、`r6yousiki (1).doc` に記載された「AI人材育成連携推進事業費補助事業計画書」に基づくものであり、地方公共団体が発行する政策評価シート及びこれに類する文書について、生成 AI を用いて市民に分かりやすいプレゼンテーションを実現するアプリケーションを開発することにある。

したがって、本事業の目的は単に PDF を読み取ることではなく、政策評価資料を機械で利用可能な形に整理し、その内容を市民、議会及び行政実務の場で活用できる形で提示することにある。また、構造化した情報を検索可能な形で公開し、継続的かつ現実的に運用できる仕組みを整えることも、当初計画の重要な要素である。

なお、事業の実施に当たっては外部要因の影響もあった。自治体へのアポイントメントについては、2026年2月に衆議院選挙が実施された影響により、各自治体との日程調整が十分に進められない時期があった。このため、自治体との対面又は個別調整を前提とした確認作業には一定の制約が生じ、事業の進行に影響を与えた。

## 2. 当初の実装方針

開発初期においては、政策評価資料を OCR により読み取り、その結果を構造化し、さらにその構造化結果を基に画面を生成するという流れを想定していた。具体的には、YomiToku を中心とした OCR 処理を用い、PDF から取得した文字情報及び表情報をもとに利用者向けの UI を構築する方針であった。この段階では、OCR の精度と表抽出の性能を高めることが、そのまま最終的な利用価値の向上につながると考えていた。

しかしながら、実データによる検証を行った結果、この前提には大きな見直しが必要であることが明らかとなった。ベンチマークとして利用した `R7【政策Ⅰ】政策推進プラン構成事業一覧表.pdf` は、画像ベースの PDF ではなく、もともと文字情報を保持している born-digital PDF であった。そのため、重い OCR 処理を前提とする設計は必ずしも適切ではなく、むしろ処理時間の増大と後段の不安定化を招く要因となっていた。

## 3. OCR 中心方針の見直し

上記の検証結果を踏まえ、born-digital PDF については OCR を主経路とせず、PDF 内部の文字情報を直接抽出する fast path を導入した。これにより、実データに対して不要な OCR を回避し、処理時間を大幅に短縮することができた。この判断自体は妥当であり、入力取得の面では一定の改善が得られた。

しかし、ここで新たに明らかになったのは、問題の中心が OCR の速度や文字認識精度ではなく、取得したデータをどのように事業単位で整理するかにあるという点である。文字そのものは取得できても、どの行が一つの事業を表しているのか、どの列が事業名でどの列が事業概要であるのか、またどの列が評価指標や数値に当たるのかを、安定的に整理できなければ、利用者にとって意味のある画面にはならない。すなわち、当初の時点では「読めること」と「使えること」を同一視していたが、実際には両者は別の課題であった。

## 4. 表 parser 中心の改善の限界

次の段階では、表の形状に応じて複数の parser を用意し、表をできるだけ安定して CSV 又は正規化テーブルへ変換する方針を採用した。markdown table 用、fixed width 用、複数ヘッダ用、予算表用などの parser を用意し、必要に応じて Gemini に parser 選択を補助させる設計とした。

この方針は、単純な表であれば有効であると考えられたが、行政の政策評価資料に含まれる表は、予算、評価指標、評価結果、今後の方向などが一つの横長の表の中に同時に含まれており、一般的な表 parser だけでは十分に整理できなかった。また、parser が成功したとしても、それによって得られるのは「整った表」であり、最終的に必要な「事業単位の意味づけされたデータ」そのものではなかった。したがって、parser の充実だけでは本事業の目標には到達しないことが明らかとなった。

## 5. Generative UI の理解不足

本開発において最も大きな反省点の一つは、`Generative UI` という考え方の実装上の捉え方が不十分であったことである。開発途中では、文書全体を要約し、主要論点や概要をカード状に表示する画面を生成していたが、これは実際には「要約 UI」に近いものであり、政策評価データを制御可能な形で提示するという本来の目的とは異なっていた。

利用者が必要としていたのは、抽出済みデータが確認できること、事業名をクリックすると当該事業の詳しい内容に移動できること、評価指標や数値が正しく表示されること、さらに政策評価の目的、手法、効果、評価といった情報の関係が追えることであった。これに対し、要約中心の画面では、見た目としてはそれらしく見えても、実際に利用可能な政策評価プレゼンテーションにはなり得なかった。この段階で、見た目を生成することと、データに基づく UI を生成することとは全く異なる課題であることを認識した。

## 6. 固定 UI への回帰とその限界

その後、要約 UI への偏りを是正するため、文書一覧、抽出済み表、案件詳細、未確定表といった固定的なセクションを持つ UI に一度戻した。これは、少なくとも恣意的な要約ではなく、抽出済みデータをそのまま確認できるという意味では一定の前進であった。

しかしながら、この方式にも限界があった。CSV の内容がほぼそのまま画面に露出し、`1-1` のような行番号や `継続` などの管理値が、誤って事業名のように見えてしまう問題が生じた。また、詳細ページへの遷移は設けたものの、表示内容が十分に整理されておらず、利用者にとっては「詳細に飛べるが、知りたい内容がきれいに出ていない」状態であった。これは、画面の問題というより、元データを UI 向けに意味づけし直す工程が不足していたことを示している。

## 7. Gemini の補助利用では不十分であったこと

次に、Gemini を列選択や表修復の補助として利用する方針を試した。例えば、どの列が事業名に当たるかを Gemini に判定させたり、parser が失敗した表に対して `llm_repair` を適用したりする構成である。しかし、この方法では、入力自体が十分に整理されていない場合に根本的な改善には至らなかった。特に、表全体の修復を Gemini に任せた場合、`r1c1` や `r1c2` のような意味のない placeholder が結果として返る場合があり、政策資料に対する実務的な精度には達しなかった。

この時点で、Gemini は補助的に使うだけでは不十分であり、むしろ「何を入力として与え、何を返させるか」を再設計する必要があることが分かった。

## 8. 構造化を Gemini の主な役割とする方針への転換

この反省を踏まえ、方針を大きく変更し、表の一部を修復するのではなく、raw CSV と layout text から、事業番号、事業名、事業概要、評価指標等を Gemini に直接 JSON として返させる方向へ移行した。これは本事業の目的に照らして、より妥当な方向であった。なぜなら、政策評価資料の価値は、見た目の表そのものよりも、そこに含まれる事業ごとの情報を、再利用可能な構造として取り出せることにあるからである。

実データを確認すると、例えば `1-7 被災地こころのケア対策事業` の行には、事業番号、事業名、事業概要のみならず、活動内容指標名、単位、計画値、実績値、成果指標名、目標値、実績値までが相当程度きれいに含まれていた。このことから、問題は OCR の不足ではなく、raw CSV を strict な JSON に変換する工程の不足にあることが明らかとなった。

さらに検証の過程で、`pdf_text_fast_path` を通過した場合には raw CSV が画面生成側に渡っていないという重大な不具合も判明した。この不具合により、Gemini には layout text と壊れた parser 結果しか渡らず、`r1c1 / r1c2` のような無意味な結果が生じていた。これを修正し、fast path の場合でも backend から raw CSV と raw JSON を補完するようにしたことで、初めて事業番号と事業名がまともに再構造化されるようになった。

## 9. 現在到達している水準

現時点では、ベンチマークとして用いている `R7【政策Ⅰ】政策推進プラン構成事業一覧表.pdf` に対して、132 件の事業を Gemini により再構造化し、一覧上で `事業番号 + 事業名` の形で表示し、詳細ページへ遷移できる状態まで改善している。実際に、`1-1 健康いわて２１プラン推進事業費`、`1-2 イー歯トーブ８０２０運動推進事業費` といった表示は成立している。

この点は、開発初期に見られた generic な要約カードや、`r1c1` のような破綻した出力と比較すれば、明確な改善である。また、事業番号と事業名の組を一覧に出し、その詳細へ移動できるようになったことは、補助事業計画書にある「市民に分かりやすいプレゼンテーションを実施するアプリケーション」という目標に向けた一歩といえる。

## 10. 現時点で残っている課題

ただし、現状はなお十分とは言えない。最も大きい課題は、評価指標の扱いである。現在の実装では `indicators[]` という単一配列で指標を保持しているが、政策評価資料においては、活動内容指標と成果指標を分けて扱う必要がある。また、それぞれについて、計画値、実績値、目標値、達成度、単位を区別して保存しなければ、利用者が本当に知りたい「何をどこまで実施し、どのような成果があったのか」という問いに十分に答えられない。

例えば、前記の `1-7 被災地こころのケア対策事業` では、活動内容指標として「岩手県こころのケアセンター運営箇所」があり、その値は 5 箇所に対して実績 5 箇所である。また、成果指標として「こころのケアケース検討数」があり、その値は 364 件に対して実績 521 件である。これらは raw row には存在しているが、現状では画面上で十分に整理されて表示されていない。したがって、今後は `activity_indicators[]` と `outcome_indicators[]` を分け、各指標について `name` `unit` `planned_value` `actual_value` `target_value` `achievement` を持つ strict なスキーマへ移行する必要がある。

また、現在の UI は依然として「Gemini で構造化した JSON を固定 UI に流し込む」段階にとどまっている。市民向け、議員向け、行政職員向けに応じて、何を前面に出し、どの順番で見せるかを変える設計にはまだ到達していない。補助事業計画書に記された「プレゼンテーション品質の向上」を達成するためには、構造化と UI 生成の間に、利用者別の表示戦略を挟む必要がある。

## 11. 現時点で得られた知見

以上の試行錯誤から得られた知見は明確である。第一に、OCR の性能向上だけでは本質的な解決には至らない。問題の中心は、読み取った文字をどのように意味のある構造へ整理するかにある。第二に、parser を増やすだけでは限界があり、最終的には事業単位の strict な JSON を中心に据える必要がある。第三に、`Generative UI` と称して要約画面を作るだけでは、本事業の目的である市民向けの分かりやすい提示にはならない。第四に、完全に共通な万能スキーマを最初から目指すのではなく、共通コアとなる項目と拡張項目を分けて設計する必要がある。第五に、`raw PDF -> 直接 UI 生成` の方式は不安定であり、`抽出 -> strict JSON -> UI 生成` の三段構成を採る方が、他自治体資料にも対応しやすい。

## 12. 今後の改善方針

今後の改善方針としては、第一に、Gemini を用いて事業単位の strict domain JSON を安定して生成する層を固める必要がある。この JSON には少なくとも `project_number`、`project_name`、`project_summary`、`activity_indicators[]`、`outcome_indicators[]`、`extra_fields` を含めるべきである。第二に、その JSON を基にして、一覧ページ、詳細ページ、評価指標カード、目標値・実績値・達成度表示といった UI を構築する必要がある。第三に、その上で初めて `Generative UI` の考え方を導入し、市民向け、議員向け、行政職員向けに応じた画面構成を生成する段階へ進むべきである。

すなわち、今後の重点は OCR そのものの改善ではなく、政策評価資料を「事業ごとの意味のある構造」に変換する工程と、その構造を利用者にとって分かりやすい形で提示する工程の設計に置かれるべきである。

## 13. 結語

本開発の経過を振り返ると、初期段階では OCR、parser、LLM、UI の役割分担を適切に整理できておらず、その結果として、見た目は動作していても実用性に乏しい状態を何度も生じさせたことが分かる。しかし、実データを用いた繰り返しの検証により、問題の所在は OCR 自体ではなく、政策評価資料を strict な構造へ変換し、それを適切な UI として提示する責務設計にあることが明確になった。

したがって、今後は補助事業計画書に記載した本来の目的に立ち返り、政策評価データの機械可読化、構造化情報の整理と公開、市民向けプレゼンテーションの実現、検索及び横断利用の仕組みの整備という一連の流れを、段階的かつ着実に実装していく必要がある。
